    // L7Rules is a union of port level rule types. Mixing of different port
// level rule types is disallowed, so exactly one of the following must be set.
// If none are specified, then no additional port level rules are applied.
type L7Rules struct {
        // HTTP specific rules.
        //
        // +optional
        HTTP []PortRuleHTTP `json:"http,omitempty"`

        // Kafka-specific rules.
        //
        // +optional
        Kafka []PortRuleKafka `json:"kafka,omitempty"`

        // DNS-specific rules.
        //
        // +optional
        DNS []PortRuleDNS `json:"dns,omitempty"`
}

    1. L7 策略的结构与规则合并

    • 规则结构是“联合”（Union）类型

      • 每条 toPorts 规则只能包含一种 L7 协议（如 HTTP 或 DNS，不能同时定义）。
      • 如果多个 toPorts 规则选择相同的 PortProtocol（如 TCP 80）并作用于同一组端点：
        • 相同类型的 L7 规则（如多个 HTTP 规则）会被合并。
        • 不同类型的 L7 规则（如 HTTP + DNS）会导致策略被拒绝（冲突）。

    • L7 规则的匹配逻辑

      • 每个 L7 规则包含一个应用协议规则列表（如 HTTP 路径或 DNS 域名）。
      • 只要满足任意一条规则，流量就会被允许（逻辑是 OR）。
      • 如果未指定任何 L7 规则，则允许所有流量（仅受 L4 端口限制）。

    2. L7 策略的特殊行为

    • 与 L4 规则的优先级
      • 如果策略中同时存在 纯 L4 规则 和 L4+L7 规则：L7 部分会被忽略，仅 L4 规则生效。

    egress:
  - toPorts:  # 纯 L4 规则（允许所有 HTTP 流量）
      - port: "80"
        protocol: TCP
  - toPorts:  # L4+L7 规则（尝试限制路径，但无效）
      - port: "80"
        protocol: TCP
        rules:
          http:
            - path: "/admin"
              method: "GET"

    最终效果是 允许所有 TCP 80 流量，/admin 限制不生效。

    • L7 违规的响应方式

      • 不同于 L3/L4 策略直接丢包，L7 策略会尝试返回协议特定的拒绝消息：
        • HTTP：返回 403 Forbidden。
        • DNS：返回 REFUSED。
        • 其他协议可能无法生成响应（依赖底层代理支持）。

    • 端口范围支持

      • L7 规则支持端口范围（如 8000-8080），但 DNS 规则除外（仅限单端口，如 UDP 53）。

    3. 主机策略（Host Policies）的限制

    • 仅支持 DNS L7 规则，如果策略使用 Node Selector（主机策略），则：
      • 只有 DNS 类型的 L7 规则有效（如限制节点访问特定域名）。
      • 其他 L7 协议（如 HTTP）无法在主机策略中使用。
    • 注意：此功能是 Beta 阶段。

    4. 实现依赖与限制

    • 依赖 Envoy 代理，L7 流量会被代理到 节点本地的 Envoy 实例：
      • Envoy 可能以 DaemonSet 或 嵌入 Cilium Agent Pod 的形式运行。
      • 风险：如果 Cilium Agent Pod 崩溃，L7 策略将失效。
    • IPv6 SNAT 流量的要求，对于 SNAT 转换的 IPv6 流量（如 Pod 访问公网）：
      • 需要特定内核版本修复支持（否则 L7 策略可能不生效）：
        • 稳定版本：6.14.1, 6.12.22, 6.6.86, 6.1.133, 5.15.180, 5.10.236。

    # HTTP

    可以匹配以下字段：

    • Path

      • Path 是一个扩展的 POSIX 正则表达式，用于匹配请求的路径。路径必须以 / 开头。如果省略或为空，则允许所有路径。

    • Method

      • Method 是一个扩展的 POSIX 正则表达式，用于匹配请求的方法，例如 GET、POST、PUT、PATCH、DELETE 等。如果省略或为空，则允许所有方法。

    • Host

      • Host 是一个扩展的 POSIX 正则表达式，用于匹配请求的 host 标头，例如 foo.com。如果省略或为空，则忽略 host 标头的值。

    • Headers

      • Headers 是一个 HTTP 标头列表，这些标头必须存在于请求中。如果省略或为空，则无论是否存在标头，请求都会被允许。 还可以对标头值进行更高级的匹配。
      • HeaderMatches 是一个 HTTP 标头列表，这些标头必须存在并与给定的值匹配。Mismatch 字段可用于指定当没有匹配项时应执行的操作。

    示例1: 允许从标签为 env=prod 的 endpoint 向标签为 app=service 的 endpoint 发送对 URL /public 的 GET 请求，但对任何其他 URL 或使用其他方法的请求都将被拒绝。除端口 80 之外的端口上的请求将被丢弃。

    apiVersion: "cilium.io/v2"
kind: CiliumNetworkPolicy
metadata:
  name: "rule1"
spec:
  description: "Allow HTTP GET /public from env=prod to app=service"
  endpointSelector:
    matchLabels:
      app: service
  ingress:
  - fromEndpoints:
    - matchLabels:
        env: prod
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
      rules:
        http:
        - method: "GET"
          path: "/public"

    apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: "tofqdn-dns-visibility"
spec:
  endpointSelector:
    matchLabels:
      any:org: alliance
  egress:
  - toEndpoints:
    - matchLabels:
       "k8s:io.kubernetes.pod.namespace": kube-system
       "k8s:k8s-app": kube-dns
    toPorts:
      - ports:
         - port: "53"
           protocol: ANY
        rules:
          dns:
            - matchName: "cilium.io"
            - matchPattern: "*.cilium.io"
            - matchPattern: "*.api.cilium.io"

  - toFQDNs:
      - matchName: "cilium.io"
      - matchName: "sub.cilium.io"
      - matchName: "service1.api.cilium.io"
      - matchPattern: "special*service.api.cilium.io"
    toPorts:
      - ports:
         - port: "80"
           protocol: TCP
