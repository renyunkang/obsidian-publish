# Pod 对外提供服务时，Cilium BGP PodCIDR 与 Multus 的取舍

# 问题定义

讨论的问题是：

  • 当 Pod 内服务需要对外提供服务时，是不是应该使用 Cilium BGP Control Plane 对外宣告 PodCIDR
  • 还是使用 Multus 给部分 Pod 增加一张额外网卡，再通过 macvlan 或 ipvlan 让这些 Pod 直接接入额外网络

这里不仅要看技术能不能跑，还要把下面这些现实约束一起纳入判断：

  • Kubernetes 管理团队和物理网络团队不是一个团队
  • BGP 一旦进入生产治理，影响面通常不只在集群内部
  • 网络团队通常会关心 ASN、peer、prefix-list、max-prefix、路由策略、变更窗口、回滚和告警归属

# 结论摘要

如果限定只在 PodCIDR 宣告 和 Multus + macvlan/ipvlan 两者里选，我更偏向：

  • 当前落地选择 Multus
  • 但只把它当成少量特殊业务的局部接入方案
  • 不建议把它当成默认的平台级对外暴露模型

更准确地说：

  • 少量特殊业务、组织边界明显、网络侧不希望 Kubernetes 团队进入动态路由治理时，优先考虑 Multus
  • 如果未来希望形成长期、标准化、平台级的对外可达能力，更合理的终局通常不是 PodCIDR + BGP，而是 Service VIP + BGP

# 为什么当前更偏向 Multus

# 1. 真正的分界点在治理边界，不在技术可行性

PodCIDR + BGP 的本质，是把 Kubernetes 集群接进物理网络的路由域。

从 Cilium 官方定义看，BGP Control Plane 负责把 Pod 网络和 Service VIP 通告给外部 BGP 邻居，但它不负责集群内部可达性的数据面编程。因此一旦出问题，排障天然跨两个团队：

  • Kubernetes 团队
  • 物理网络团队

如果组织上没有先明确这些问题，技术上能跑，运维上也会很脆：

  • 谁负责建邻和变更
  • 谁维护 ASN、peer、prefix allow-list
  • 谁设置 max-prefix
  • 谁处理路由收敛异常
  • 谁是事故 owner

# 2. PodCIDR 广告解决的是 Pod 前缀可达，不是稳定服务身份

这个点很重要。

Cilium 官方支持两类与你当前问题最相关的广告对象：

  • PodCIDR
  • Service VIP

其中：

  • PodCIDR 广告的是节点本地持有的 Pod 前缀
  • Service VIP 广告的是精确 VIP 路由，通常是 /32 或 /128

所以如果问题表述为“Pod 内服务需要对外提供服务”，从架构抽象上更接近的目标其实是：

  • 让外部访问一个稳定的服务地址

而不是：

  • 让外部网络可路由到所有 Pod 地址

也就是说，PodCIDR + BGP 更像“Pod 网络暴露”，不是最自然的“服务发布”模型。

# 3. Multus 的影响域更容易收敛

Multus 的官方 Quickstart 很直白：Pod 在默认 eth0 之外，再挂一张额外接口，例如 net1。

这意味着你可以把需求限定在一个更小的边界里：

  • 只有少数 namespace 使用
  • 只有少数节点承载
  • 只接一个专用 VLAN 或独立子网
  • 只让网络团队提供二层或三层边界条件

这样做的结果是：

  • 不需要把整个集群纳入动态路由治理
  • 不需要把 Kubernetes 团队变成 BGP 邻居管理者
  • 审批链通常更短
  • 出问题时影响面更局部

# 但为什么我不把 Multus 当终局

Multus 更像附加网络机制，而不是统一的服务暴露平面。

一旦它从“少量特殊业务”演变成“大家都这么干”，后面通常会出现这些问题：

  • 每类业务都想要自己的子网或 VLAN
  • IPAM 逐渐碎片化
  • 调度约束开始和网络位置强耦合
  • 额外网卡、路由、DNS、ACL、监控、排障路径越来越不一致

所以我会把它定义为：

  • 战术方案

而不是：

  • 平台默认方案

# 对 PodCIDR + BGP 的修正理解

之前一个容易说粗的点，需要明确修正。

# 1. 它不是把整个集群 Pod 大网段一次性整体广播出去

按 Cilium 官方文档，在使用 Kubernetes 或 ClusterPool IPAM 时，advertisementType: "PodCIDR" 的语义是：

  • 每个节点上的 BGP 实例对外通告“分配给本节点的 PodCIDR 前缀”

因此它的精确含义是：

  • Node A 宣告 Node A 的 PodCIDR
  • Node B 宣告 Node B 的 PodCIDR

不是单点把整个集群 Pod 地址池整体打进外部网络。

# 2. 但它仍然是平台级能力，不是小范围开口子

即便如此，对网络团队来说，这仍然意味着：

  • 物理网络需要接受来自集群节点的动态路由
  • 需要审查可接受前缀
  • 需要控制路由数量与策略边界

所以它仍然更像“平台网络能力建设”，而不是“给几个业务开访问”的局部方案。

# 如果走 Multus，优先评估 ipvlan，不要默认 macvlan

这里建议再压实一点。

# macvlan 的优点

官方文档说明：

  • macvlan 会在共享父接口的前提下，为每个虚拟接口分配独立 MAC 地址
  • 这让它更容易和现有 DHCP 体系配合

# macvlan 的代价

它也意味着：

  • 网络侧会看到更多 MAC
  • 对交换网络的 MAC 学习、接入口策略、运维习惯更敏感

# ipvlan 的特点

官方文档说明：

  • ipvlan 和父接口共享同一个 MAC
  • 内核按 IP 决定流量进入哪个虚拟接口

这通常意味着：

  • 对网络设备更友好
  • 更适合网络团队对 MAC 数量敏感的环境

但也有一个明确限制：

  • 由于共享 MAC，DHCP 只有配合 ClientID 才可能工作，而官方文档说明当前 DHCP plugin 不支持这一点

因此实际建议是：

  • 你如果主要追求“少量业务直接接入外部网段”，优先评估 ipvlan
  • 只有在明确依赖 DHCP 习惯或特定二层行为时，再优先考虑 macvlan

# 选择建议

# 适合选 Multus 的场景

  • 只有少量特殊业务需要直接对外提供网络入口
  • 这些业务可以接受额外的网络模型和运维约束
  • 网络团队不希望 Kubernetes 团队成为正式 BGP 邻居
  • 变更窗口严格，希望把影响面锁在少数节点和少数网段内

# 适合推动 BGP 的场景

  • 需求不是个别业务，而是平台级共性能力
  • 网络团队接受集群作为正式 BGP 邻居
  • 双方对路由治理边界有明确共识
  • 你准备把它做成标准能力而不是例外流程

# 但即使推动 BGP，也不建议把 PodCIDR 宣告当终局

如果目标真的是“服务对外发布”，更合理的演进方向通常是：

  • 优先做 Service VIP 广告

而不是长期停在：

  • PodCIDR 暴露

原因很简单：

  • Service VIP 更贴近服务语义
  • Service VIP 粒度更精确
  • Cilium 官方也明确支持对 Service 虚拟 IP 做精确路由广告

# 一句话判断

  • 少量特殊业务、团队边界明显、网络变更敏感：选 Multus，并优先评估 ipvlan
  • 想做长期、平台级、标准化的对外可达能力：推动 BGP，但目标应更偏向 Service VIP，而不是长期暴露 PodCIDR

# 如果一定要上 PodCIDR + BGP，前置条件至少要满足

  • 物理网络团队明确接受 Kubernetes 节点作为正式 BGP 邻居
  • 有清晰的 prefix allow-list，只允许集群应通告的前缀
  • 有 max-prefix、告警、回滚和变更窗口
  • 明确 incident owner，不能在故障时再临时分责
  • 最好放在独立边界层或 VRF 中，而不是直接进入核心共享路由域

# 补充判断

如果问题的真实需求不是“让外部路由到 Pod”，而是“让外部稳定访问某个服务”，那么更准确的优先级通常应该是：

  1. 先看能否用 Service VIP 模型解决
  2. 组织上推不动 BGP 时，再用 Multus 为少量业务做局部接入
  3. 不把 PodCIDR + BGP 当成默认服务发布方案
