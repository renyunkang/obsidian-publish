    #calico #issue

    • 配置etcd calico 支持两种数据存储，一种是使用 kubernetes api 接口来对其 etcd 进行读写操作；一种直接连接 etcd 进行读写操作，同时 calico 支持配置 etcd tls 进行数据加密实现安全的存储，对于 etcd 相关的证书生成以及更新维护不在 calico 管理范围之内。

    docs/generate-self-signed-certificates.md at master · coreos/docs (github.com) calico 通过 configmap 用于 daemonset 配置环境变量 + Secret 配置证书 base64 编码 对于 etcd 的证书要与集群的证书进行统一管理

    • 扩容、新增 ippool 使用 kk 安装集群时会使用 kubePodsCIDR: 10.233.64.0/18 创建一个默认子网，此时创建的工作负载会使用该 ippool 来分配 ip，如果该 ippool 地址分配满而不够使用时，可以新建一个 ippool 使用新的 ippool 来分配 ip。

      一般情况不进行 ippool 扩容，因为扩容操作比较繁琐而且可以通过新建ippool 的方式避免。如果有必要可以通过以下操作进行扩容：需要先创建一个临时的 ippool 并禁用要扩容的 ippool，将原有 ippool 的业务 pod 删除之后在新的 ippool 重新分配，删除并重建扩容的 ippool，再将临时创建的 ippool 禁用并将业务 pod 删除，pod 便会在新的 ippool 重新分配，完成操作之后将临时的 ippool 删除

    • calico 升级 对于升级过程中，最快的升级类比于重启calico daemonset， 影响的时间需要评估

    部署：

    • 部署时的网络规划 方案确定：underlay or overlay overlay 模式不关心底层物理网络，使用 ipip 封装或者 vxlan 封装进行包的转发 underlay 依赖于底层的物理网络，部署方式以及考虑的情况也取决要部署环境的网络架构

    • calicoctl 工具部署 尽量在全物理节点部署，方便在各个节点上单独维护以及排查问题

    • kubekey 支持 calico 配置选项

      • pod cidr - 默认 ippool cidr
      • ipip/vxlan mode，veth mtu 在安装完成后手动完成 bgp rr 模式相关的配置

    • calico 选择 RR 节点

    1. 选取 kubernetes 节点当做 rr 节点 需要在选取的 RR 节点上打 label，之后设置 routeReflectorClusterID(该 id 表示 BGP 路由器的集群 id，id 唯一即可没有特殊的要求，但是为了方便维护尽量取一个与所搭建集群相关的 ip 地址)，如果在一个集群下有多个 rr 节点，可以将 rr 节点的 routeReflectorClusterID 设置为同一个 id 建立 bgppeer 完成，rr 节点与其他节点 以及 rr 节点与交换机的 bgp 连接
    2. 选取 tor 并开启 rr 模式 在交换机上完成对应 rr 的配置，在 calico 中配置 bgpeer 即可

    Troubleshooting and diagnostics | Calico Documentation (tigera.io) Linux 系统上的一个常见问题是 conntrack 表中的空间不足，这会导致 iptables 性能不佳。如果主机上运行大量工作，或者负载会创建大量 TCP 连接或双向 UDP 流，就会发生这种情况。为避免该情况问题，使用以下命令来增加 conntrack 表的大小：

    1
2

    sysctl -w net.netfilter.nf_conntrack_max=1000000  
echo "net.netfilter.nf_conntrack_max=1000000" >> /etc/sysctl.conf
