    IPV6 支持 cilium 最初设计时是只支持 ipv6，但是随着 ipv6 的普及和客户需求，不得不加上 ipv4 和 双栈（过渡阶段）；Kubernetes 1.23 双栈 GA。

    • cilium 1.1

      • 必须 Kubernetes PodCIDR 分配参数，新增 agent options –k8s-require-ipv4-pod-cidr 和 –k8s-require-ipv6-pod-cidr，要求 Kubernetes 通过 Node 资源提供 Kubernetes PodCIDR。
      • 新增 –ipv6-cluster-alloc-cidr options，用于在 Cilium 为每个节点分配 IPv6 CIDR 时指定 IPv6 CIDR。

    • cilium 1.4

      • 安装默认禁用 ipv6，通过 --enable-ipv6=true 启用
      • ipv4 和 ipv6 Transparent encryption (beta)

    • cilium 1.7

      • 支持基于套接字的负载均衡的 IPv4-in-IPv6。Cilium 的 eBPF 中基于套接字的负载均衡功能现在也支持 IPv4 到 IPv6 的服务地址转换。这使得仅支持 IPv6 的应用程序可以在 connect(2)/sendmsg(2) 钩子函数中选择 IPv4 服务后端。在 Java 等语言运行时环境中，此功能通常默认启用。

    • cilium 1.12
      • 支持 cilium bgp ipv6 - 使用了 gobgp
      • 引入负载均衡器 NAT46/64，cilium L4LB 支持 service 的 NAT46 和 NAT64 编码。IPv4 服务 IP 地址暴露仅支持 IPv6 的 Pod，反之亦然。

    由于网络协议栈的复杂性以及与数据包绑定的地址族特定元数据（例如 GRO/GSO 相关信息，难以在两者之间迁移），NAT46 和 NAT64 从未被纳入 Linux 内核的 iptables/netfilter 子系统。鉴于 Linux 内核中的 eBPF 模型更为简单，我们通过 bpf_skb_change_proto() 辅助函数在 Linux 内核的 tc 层实现了对 eBPF 的 NAT46 和 NAT64 支持

    • cilium 1.13
      • 支持 cilium LB ipv6 地址 bgp 对外宣告（ipv4 + ipv6 同时支持）
      • 增强 NAT46/64 - 借助该功能可以实现，部署 ipv6-only 的集群访问 ipv4-only 的系统（NAT64），或者支持从 ipv4-only 的 client 访问（NAT46）。
      • 基于 IPv6 的分段路由（SRv6）（企业版）
      • L3/L4 网络可观测性和 IPv6 支持（企业版）~ Tetragon 基于socket 进行信息的收集

    NAT64 需要 DNS64。当 IPv6 客户端查找 AAAA 记录但找不到该记录时，我们可以利用 DNS64 服务器（例如 Google 公共服务器或 coreDNS）来获得满意的 DNS 解析结果

nicovibert:~$ nslookup -query=AAAA twitter.com 2001:4860:4860::6464
Server:		2001:4860:4860::6464
Address:	2001:4860:4860::6464#53

Non-authoritative answer:
Name:	twitter.com
Address: 64:ff9b::68f4:2ac1

     64:ff9b::68f4:2ac1 IPv6 地址是什么：DNS64 服务器从 DNS A 权威服务器接收 IPv4 地址，并通过在其地址前加上 NAT64 前缀 (64:ff9bb:) 来合成 AAAA 记录，并将 IPv4 地址转换为十六进制（104 的十六进制为 0x68，244 的十六进制为 0xF4，42 的十六进制为 0x2A，193 的十六进制为 0xC1）。   当 IPv6 客户端向 64:ff9b::68f4:2ac1 发送的流量到达 Cilium NAT64 网关时，Cilium 会自动将目标 IP 地址转换为原始 IPv4 地址 104.244.42.193。

     

    • cilium 1.18
      • 此前，Cilium overlay 模式一直基于 IPv4 底层网络。从 Cilium 1.18 开始，overlay 模式（VXLAN 或 Geneve）支持 IPv6 底层网络。
      • KPR 在之前版本假定底层网络为 IPv4。从 Cilium 1.18 开始，KPR 支持 IPv6 底层网络上的服务转换。
      • Cilium 1.18 中 Egress Gateway Policy支持匹配 IPv6 目标 CIDR。这使得源 IP 地址保留和出口控制功能扩展到 IPv6 工作负载，从而实现了与 IPv4 相同的控制级别和灵活性。
