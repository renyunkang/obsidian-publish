    # Overlay 安装

    overlay 的安装可以满足大部分容器网络需求，建议首选。

    默认安装所选，其对底层网络基础设施的要求最低。 在此模式下，所有集群节点使用基于 UDP 的封装协议 VXLAN 或 Geneve 形成隧道网。Cilium 节点之间的所有流量都会被封装。

    • VXLAN (Default) 8472/UDP
    • Geneve 6081/UDP

    简洁性 连接集群节点的网络无需感知 PodCIDR。集群节点可以生成多个路由或链路层域。只要集群节点能够通过 IP/UDP 相互通信，底层网络的拓扑结构就无关紧要。

    寻址空间 由于不依赖于任何底层网络限制，可用的寻址空间可能更大，并且如果 PodCIDR 大小配置得当，则允许每个节点运行任意数量的 Pod。

    自动配置 当与 Kubernetes 等编排系统一起运行时，集群中所有节点的列表（包括其关联的分配前缀节点）会自动提供给每个代理。新加入集群的节点将自动集成到网格中。

    身份上下文 封装协议允许将元数据与网络数据包一起传输。Cilium 利用此功能传输元数据，例如源安全身份。身份传输是一种优化，旨在避免在远程节点上进行一次身份查找。

    缺点： 由于添加了封装头，有效载荷可用的有效 MTU 比原生路由（VXLAN 每个网络数据包 50 字节）要低。这导致特定网络连接的最大吞吐量降低。启用巨型帧可以很大程度上缓解这个问题（每 1500 字节增加 50 字节的开销，而每 9000 字节增加 50 字节的开销）。

    # Underlay 安装

    当集群无法使用或不想使用 vxlan/Geneve udp 封装时，可以使用 cilium native 模式，利用 Cilium 运行所在网络的路由功能，而不是进行数据包封装。

    在原生路由模式下，Cilium 会将所有未发往其他本地端点的数据包委托给 Linux 内核的路由子系统。这意味着数据包的路由方式与本地进程发出数据包的方式相同。因此，连接集群节点的网络必须能够路由 PodCIDR（连接运行 Cilium 的主机的网络必须能够使用分配给 pod 或其他工作负载的地址转发 IP 流量）。

    节点上的 Linux 内核必须知道如何转发运行 Cilium 的所有节点的 Pod 或其他工作负载的数据包。这可以通过两种方式实现：

    • 节点本身并不知道如何路由所有 Pod 的 IP 地址，但网络中存在一个路由器，它知道如何访问所有其他 Pod。在这种情况下，Linux 节点配置了一条默认路由，指向该路由器。此模型用于云提供商网络集成。

    • 每个节点都会获知所有其他节点的 Pod IP 地址，并将相应的路由信息​​添加到 Linux 内核路由表中。

      • 如果所有节点共享同一个 L2 网络，则可以通过启用 auto-direct-node-routes: true 选项来实现。
      • 如果所有节点不在同一个 L2 网络，需要运行额外的系统组件（例如 BGP 守护进程）来分发路由。

    如果需要将 pod ip 提供给外部直接访问，可以使用 cilium bgp 将 pod cidr 向外部宣告，也可以使用 multus-cni + macvlan 打通底层物理网络的方式
