    # Entities based

    有以下 Entity：

    EntityWhat it selectsUse case
    cluster集群中的 Pods and nodesAllow broad east–west communication as a baseline
    world集群外的所有Coarse egress control.
    kube-apiserver控制平面 endpointEnsure workloads can talk to the API server if needed
    hostThe local node控制 node-local daemons or host networking pods
    remote-node集群其他节点 nodeSpecial cases like node-to-node probes
    ingress, healthCilium ingress, health checksPermit Cilium’s own internal components
    • init：
      • init entity 包含所有处于引导阶段但尚未解析安全身份的端点。这种情况通常只在非 Kubernetes 环境中出现。
    • unmanaged：
      • 非托管实体代表 Cilium未管理的 endpoint 。非托管端点被视为 cluster 的一部分，并包含在cluster entity 中。
    • all：
      • all 代表所有已知 clusters 以及 world 的组合，并允许所有通信。

    在某些 Kubernetes 发行版（例如 Azure AKS 和 GCP GKE）中，kube-apiserver 实体可能无法用于入口流量。这是因为入口控制平面流量会通过工作节点进行隧道传输，从而无法保留原始源 IP 地址。可以尝试使用范围更广的 fromEntities: cluster 规则。

    通过 toEntities: kube-apiserver 限制出口流量应该可以在这些 Kubernetes 发行版中正常工作。

    # Node based

    下面的示例（包含 fromNodes/toNodes 字段）仅在 enable-node-selector-labels 标志设置为 true（或等效的 Helm 值 nodeSelectorLabels: true）时才会生效。

    默认情况下，cilium 会自动从 cluster mesh 环境中的所有集群中选择节点。可以通过配置将节点选择限制为本地集群。通过 ConfigMap 选项 policy-default-local-cluster 或 Helm 值 clustermesh.policyDefaultLocalCluster 启用 –policy-default-local-cluster 选项。

    当指定 --enable-node-selector-labels=true 时，每个 cilium-agent 都会为所有其他节点分配不同的本地安全身份。但它不会使用本地作用域的身份，而是使用远程节点作用域的身份范围。

    允许所有带有 env=prod 标签的 endpoint 仅从 cluster（或 cluster mesh）中的控制平面（label 为 node-role.kubernetes.io/control-plane="") 节点接收流量。

    # IP/CIDR based

    CIDR 策略用于定义 与未被 Cilium 管理的端点（无标签的实体）之间的通信规则，包括：

    • 外部服务（如公网 API、数据库）。
    • 非 Kubernetes 管理的虚拟机（VMs）或物理机。
    • 集群外部的网络设备（如传统数据中心子网）。

    CIDR 策略可以应用于i ngress 或 egress。 若无法将通信的源或目标 IP 映射到基于标签的端点身份，则 CIDR 规则适用。

    不适用 CIDR 规则的流量：

    • 两端均为 Cilium 管理的端点（如两个 Pod 通信）。
      此类流量应通过 标签（Labels）、服务（Services）或实体（Entities）策略 控制。
    • 两端属于集群节点 IP（包括 HostNetwork Pod）。
      除非显式配置，否则默认由其他策略（如 toEntities: host）管理。

    # Ingress

    fromCIDR

    • 允许与 endpointSelector 选择的所有 endpoint 通信的源 prefixes/CIDRs 列表。

    fromCIDRSet

    • 允许与 endpointSelector 选定的所有 endpoint 通信的源 prefixes/CIDR，以及每个源 prefixes/CIDR 的子网列表（可选），这些子网不允许进行通信。
    • fromCIDRSet 还可以通过 CiliumCIDRGroup 间接引用 prefixes/CIDR。

    # Egress

    toCIDR

    • endpointSelector 选择的 endpoint 可以与之通信的目标前缀/CIDR 列表。
      • 自动允许回包：如果流量是通过 fromEndpoints 规则触发的入站请求的响应包（如 TCP 回包），则无需显式配置，Cilium 会自动放行。

    toCIDRSet

    • 与 toCIDR 类似，但支持 更精细的控制：允许定义目标 CIDR 的同时，还可以指定该 CIDR 范围内的 except 子网（不允许访问的子网）。

    # 选择具有 CIDR/ipBlock 的节点 - beta feature

    默认 CIDR 选择器不匹配集群内实体，但是可以通过配置 --policy-cidr-match-mode=nodes（或在 Helm 中设置 policyCIDRMatchMode: nodes），可以强制 CIDR 选择器覆盖集群节点 IP

    #  自托管集群（Self-hosted Cluster）的特殊需求

    • 场景：当 Kubernetes 的 kube-apiserver 运行在集群节点上时（非外部托管），默认的 CIDR 规则无法直接匹配其节点 IP。
    • 传统解决方案：
      需在 CiliumNetworkPolicy 中使用 toEntities: kube-apiserver 来放行流量。
    • 启用 nodes 模式后的解决方案：
      可直接在 KubernetesNetworkPolicy（或 CiliumNetworkPolicy）中用 ipBlock 选择 kube-apiserver 的节点 IP。

    # 关键注意事项：

    • 性能影响：
      启用 nodes 模式后，CIDR 规则会额外匹配节点 IP，可能增加策略匹配的复杂度。
    • 安全性：
      需确保 CIDR 规则中的节点 IP 范围精确，避免意外放行其他节点流量。
    • 替代方案：
      如果仅需控制 kube-apiserver 访问，优先使用 toEntities: kube-apiserver（无需启用此模式）。

    # DNS based

    DNS 策略用于为 未被 Cilium 管理但具有 DNS 域名 的外部端点（如云服务、数据库）定义 L3 网络策略。类似于 CIDR 策略，但基于 DNS 响应动态更新允许的 IP 列表。

    适用场景：

    • 外部服务的 IP 地址可能动态变化（如轮询 DNS）。
    • 使用域名比直接指定 CIDR 更灵活（例如 api.example.com）

    实现机制

    • Cilium 通过 每节点部署的 DNS 代理 拦截 DNS 响应，获取域名对应的 IP 地址。
    • 依赖条件：
      • 需启用 L7 代理：--enable-l7-proxy=true。
      • 需配置 L7 策略放行 DNS 请求（如允许 port 53 的 UDP/TCP 流量）。

    规则生成

    • 每个 toFQDNs 规则会动态生成 L3 CIDR 规则，允许 DNS 响应中的 IP 地址。
    • 支持两种匹配方式：
    • toFQDNs.matchName**
      • 精确匹配域名，支持多个独立条目。
    1
2

    toFQDNs:
 - matchName: "api.example.com" # 允许该域名的所有解析 IP
    • toFQDNs.matchPattern**
      • 通配符匹配域名，规则如下：
        • *：匹配任意字符（不含 .），如 *.example.com 匹配 a.example.com，但不匹配 a.b.example.com。
        • **：跨级匹配子域，如 **.example.com 匹配 a.b.example.com。
        • * 单独使用：匹配所有域名（允许所有 DNS 缓存中的 IP）。
    1
2

    toFQDNs:
 - matchPattern: "*.example.com" # 允许所有子域名

    规则限制

    • L3 规则独占性：
      toFQDNs 规则不能与其他 L3 规则（如 toEndpoints、toCIDR）混合使用。
    • 可搭配 L4/L7 规则：
      可结合 toPorts 指定端口，或添加 HTTP/Kafka 等 L7 规则。

    # 管理短生命周期连接和每个 FQDN/端点的最大 IP 地址

    1. 问题背景
    • 短生命周期连接：
      某些应用（如微服务、爬虫）会频繁建立短时连接（如 HTTP 请求），导致 DNS 解析的 IP 地址快速积累。
    • 风险：
      如果不对 IP 缓存数量进行限制，可能导致：
      • 内存占用过高（每个 IP 需存储策略规则）。
      • 策略匹配性能下降（需检查大量 IP）。
    1. Cilium 通过以下机制控制每个 FQDN 关联的 IP 数量：

    • 默认限制

      • 每个 FQDN 在每个端点（Pod）上最多缓存 50 个 IP 地址（通过 DNS 解析获取）。
      • 超出限制时，最旧的未使用的 IP 会被自动移除（LRU 策略）。

    • 调整限制

      • 可通过 --tofqdns-endpoint-max-ip-per-hostname 参数修改默认值（例如调整为 100）：
      • cilium-agent --tofqdns-endpoint-max-ip-per-hostname=100

    即使 IP 数量超出限制，正在使用的 IP 不会被立即移除，直到连接关闭。
