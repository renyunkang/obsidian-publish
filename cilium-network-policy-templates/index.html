    # Cilium NetworkPolicy 模板集

    这份模板集配合主手册一起使用:

    • 主手册: docs/cilium-network-policy-safe-rollout.md

    所有模板都需要你按实际环境替换占位符。不要直接无脑套用。

    # 占位符说明

    • <namespace>: 目标命名空间
    • <src-namespace>: 来源命名空间
    • <app-label-key> / <app-label-value>: 目标 Pod 标签
    • <src-label-key> / <src-label-value>: 来源 Pod 标签
    • <dst-port>: 目标端口
    • <fqdn>: 外部域名
    • <cidr>: 外部网段

    # 模板 1: 仅观测 DNS，不触发默认拒绝

    适用场景:

    • 你还在摸底期
    • 你想让 Cilium 看到 DNS 流量
    • 你还不想因为第一条策略就进入默认拒绝

    注意:

    • 这个模板里的 enableDefaultDeny: false 只适合 L3/L4 观测思路。
    • 官方文档明确说明，这个字段不适用于 L7 规则带来的所有拦截场景。

    apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: dns-observe-no-deny
  namespace: <namespace>
spec:
  enableDefaultDeny:
    ingress: false
    egress: false
  endpointSelector: {}
  egress:
    - toEndpoints:
        - matchLabels:
            "k8s:io.kubernetes.pod.namespace": kube-system
            "k8s:k8s-app": kube-dns
      toPorts:
        - ports:
            - port: "53"
              protocol: UDP
            - port: "53"
              protocol: TCP
          rules:
            dns:
              - matchPattern: "*"

    # 模板 3: 放行 DNS 到 kube-dns

    适用场景:

    • 应用需要解析域名
    • 后续要使用 toFQDNs

    说明:

    • 如果你的集群 DNS Pod 标签不是 k8s-app: kube-dns，请先改成实际标签。
    • 某些环境里你需要把 k8s:k8s-app 改成实际可见标签，例如 k8s:k8s-app: coredns。

    # 模板 6: 允许来自另一个 namespace 的来源访问

    适用场景:

    • 来源服务和目标服务不在同一个 namespace
    • 你要显式控制跨 namespace 调用

    说明:

    • 不要再用已经废弃的 fromRequires / toRequires 思路。
    • 直接在 fromEndpoints 里同时写来源 namespace 和来源标签。

    # 模板 10: HTTP L7 限制

    适用场景:

    • 你确实需要按 HTTP 方法和路径收紧
    • 目标端口明确就是 HTTP

    不适用:

    • TCP 直连服务
    • gRPC 以外的非 HTTP 协议
    • 还没把 L3/L4 先理清的情况

    apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-http-paths
  namespace: <namespace>
spec:
  endpointSelector:
    matchLabels:
      <app-label-key>: <app-label-value>
  ingress:
    - fromEndpoints:
        - matchLabels:
            <src-label-key>: <src-label-value>
      toPorts:
        - ports:
            - port: "<dst-port>"
              protocol: TCP
          rules:
            http:
              - method: GET
                path: ^/health$
              - method: POST
                path: ^/orders$

    # 模板 11: 显式默认拒绝某个工作负载

    适用场景:

    • 你已经补齐了必要 allow
    • 你想明确把目标工作负载收紧成默认拒绝

    说明:

    • 在 Cilium 里，只要策略选中了 Pod，并且包含 ingress 或 egress 段，该方向就会进入默认拒绝。
    • 下面这个模板是最直观的“先拒绝，再补 allow”写法。

    # 套用顺序建议

    最稳妥的组合通常是:

    1. 模板 2: 命名空间内互通基线
    2. 模板 3: DNS
    3. 模板 4: kube-apiserver
    4. 模板 5 或模板 6: 业务依赖
    5. 模板 7、模板 8 或模板 9: 出向依赖
    6. 模板 10: 必要时再加 L7
    7. 模板 11: 最后进入显式默认拒绝

    # 使用前自检

    应用前至少确认:

    • 目标 Pod 的标签没有写错
    • DNS Pod 的标签和你集群一致
    • 端口协议和实际服务一致
    • 如果写了 toFQDNs，DNS 已经先放行
    • 如果写了 L7，目标端口协议真的可被代理解析
