    # client-go 中的三个队列

    • 去重队列
    • 延时队列
    • 限频队列

    其中延时队列是基于去重队列实现，限频队列是基于延时队列实现。

    • Interface： 实现了基本的先进先出队列, 跟常规队列相比多了去重功能.
    • DelayingInterface：在 Interface 的基础上, 实现了延迟队里功能.
    • RateLimitingInterface： 又在 DelayingInterface 的基础上, 实现了 RateLimiter 限频器功能. 当插入元素的次数超过限频器规则时, 则把对象推到延迟队列中处理.

    去重队列实现原理： 其中包含：

    • 一个 FIFO 切片，表示等待处理的数据
    • 一个数据的 setA，表示队列中含有的元素
    • 一个数据的 setB，表示正在处理的元素，避免并发处理

    当元素要 add 到队列时，先看 setA 中是否存在，没有时添加，看 setB 中是否存在，如果不存在则添加到待处理队列中，如果存在则等调用 done 之后再将其添加到队列中。

    延时队列实现原理： 延时队列使用一个小根堆来对要添加到队列的数据进行存放，当要存放时循环弹出小根堆中已经达到预定时间的堆顶元素，当堆顶元素没有达到预定时间时，便设置一个数值为相差时间的定时器等待

    限频队列实现原理： 限频队列通过限频器计算出需要等待的时长, 然后调用 delayingQueue.AddAfter() 方法来决定把对象扔到延迟队里还是队列里.

    下面是 workqueue 内置的几个 RateLimiter 限频器.

    • 令牌桶限速器 BucketRateLimiter
      • 通过 rate.Limiter 进行限速，通过 rate 获取新元素需要等待的时间
    • 基于 backoff 的限速器 ItemExponentialFailureRateLimiter
      • 使用一个 map 记录了各个元素的计数, 后通过经典 backoff 算法可以求出当前需要等待的时长. 默认为 1, 只要不 Forget 抹掉计数, 那么下次再入队时, 其等待的时长为上次的二次方.
    • ItemFastSlowRateLimiter
      • 超过阈值则使用 fastDelay, 否则使用 slowDelay 等待间隔.
    • MaxOfRateLimiter
      •  可以同时对多个 rateLimiter 实例进行计算, 最后用最大的时长.

    # informer 机制

    client-go 中有 informer、sharedIndexInformer、sharedInformerFactory

    其中 Informer 是最基础的，基于内部的组件共同实现其功能，其中组件包括：

    • reflector 反射器：全责 list/watch 数据，并将增量数据放到 deltaFIFO 队列中
    • deltaFIFO 存放增量变动的队列
    • storeIndex：存放索引，加速数据的检索
    • threadSafeMap：本地缓存，用于存放数据
    • controller 控制器：实例化并启动 reflector，并调用 processLoop 来消费 deltaFifo 队列

    sharedIndexInformer 是共享reflector 反射器和 store ，业务可以注册多个 resourceEventHandler 方法, 无需重复创建 informer 做监听及事件注册. 避免同一个资源的不同 informer/listeners 重复存储数据 - （举例：多个 controller 同时监听 pods，使用一个 pod sharedIndexInformer，创建不同的监听器以及处理事件函数）

    下面是 shareIndexInformer

    HandleDeltas 用来处理从 DeltaFIFO 拿到的 deltas 事件列表, 然后通知给所有的 lisenter 去处理.

    deltas 事件：Added、Updated、Deleted、Replaced、Sync。 list-watch 中 list 往队列写的 delta 类型是 Replaced

    • 当 delta 事件类型为 sync/replaced/added/updated 时, 判断在 store 是否存在该对象, 存在则执行 store.update 操作和 OnUpdate 回调, 反之则执行 store.add 和 OnAdd 回调.
    • 当 delta 事件类型为 deleted 时, 则执行 store.delete 方法. 执行注册方法的 OnDelete. distribute 收到变更的事件后, 遍历通知给所有的 listener 监听器, 这里的通知是把事件写到 listener 的 addCh 通道.

    在添加 listener 监听器时，会启动两个协程处理 run 和 pop 方法

    • pop() 监听 addCh 队列把 notification 对象扔到 nextCh 管道里.
    • run() 对 nextCh 进行监听, 然后根据不同类型调用不同的 ResourceEventHandler 方法.

    sharedInformerFactory 组合了多个不同资源的 sharedIndexInformer 对象
