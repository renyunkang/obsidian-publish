    # 策略执行模式

    策略执行模式有以下三种，可以通过 helm values policyEnforcementMode 配置：

    • default 默认行为。endpoint 默认不受网络访问的限制。被策略选中后，endpoint 仅放行允许方向的流量通过。

    • always 在始终模式下，即使没有规则选择特定端点，所有端点都会启用策略执行。 如果您希望配置 health entity，以便在使用 enable-policy: always 启动 cilium-agent 时检查集群范围内的连接性，则可能需要启用与 health entity 之间的通信。

    # 将健康实体添加到所有 Cilium 管理的端点，以便检查集群连接健康状况。
apiVersion: "cilium.io/v2"
kind: CiliumClusterwideNetworkPolicy
metadata:
  name: "cilium-health-checks"
spec:
  endpointSelector:
    matchLabels:
      'reserved:health': ''
  ingress:
    - fromEntities:
      - remote-node
  egress:
    - toEntities:
      - remote-node
    • never 此模式下，允许来自任何 ingress 或 egress 的流量。所有 endpoint 都会禁用策略执行，即使规则选择了特定 endpoint。

    # Endpoint 默认策略

    默认情况下，所有 endpoints 的 ingress 和 egress 流量都是被允许的，当一个 endpoint 被 networkpolicy 选中后会变为 default-deny 状态，仅放行允许通过的流量。

    该 default-deny 状态也是按照方向决定：

    • 如果 rule 选中 endpoint 且该 rule 有 ingress 定义，则该 endpoint ingress 进入默认 deny-mode。
    • 如果 rule 选中 endpoint 且该 rule 有 egress 定义，则该 endpoint egress 进入默认 deny-mode。

    rule 中有 EnableDefaultDeny 字段，可以设置禁用默认拒绝模式。EnableDefaultDeny不适用于 layer-7 策略。

    以下策略会拦截整个集群的 dns 流量，但是不会阻止任何流量。管理员可以安全地在整个集群范围内应用此策略，而无需担心它会将某个端点转换为默认拒绝模式，丢弃合法流量。

    apiVersion: cilium.io/v2
kind: CiliumClusterwideNetworkPolicy
metadata:
  name: intercept-all-dns
spec:
  endpointSelector:
    matchExpressions:
      - key: "io.kubernetes.pod.namespace"
        operator: "NotIn"
        values:
        - "kube-system"
      - key: "k8s-app"
        operator: "NotIn"
        values:
        - kube-dns
  enableDefaultDeny:
    egress: false
    ingress: false
  egress:
    - toEndpoints:
        - matchLabels:
            io.kubernetes.pod.namespace: kube-system
            k8s-app: kube-dns
      toPorts:
        - ports:
          - port: "53"
            protocol: TCP
          - port: "53"
            protocol: UDP
          rules:
            dns:
              - matchPattern: "*"

    # 策略拒绝响应的处理 - 实验性

    可以通过 –policy-deny-response 选项配置 pod 流量被选中之后的 cilium 拒绝响应的回复，默认情况下当 networkpolicy 禁止 Pod 发出 egress 流量时，Cilium 会丢弃这些数据包。实际的业务在访问这些被禁止的 pod 时，会报错连接超时，而不是连接失败。

    然而，有些应用程序或许更适合接收明确的连接拒绝，而不是连接超时。也可以为应用程序提供更快的反馈，并通过减少等待时间来改善用户体验。

    –policy-deny-response 支持以下选项配置：

    • none - 默认
      • 丢弃被拒绝的数据包
    • icmp - 实验性
      • 当策略拒绝 egress 流量时，向源 Pod 发送 ICMP 目标不可达响应

    实验性功能，目前仅适用于网络策略拒绝的 IPv4 出口 Pod 流量。入口流量拒绝行为和 IPv6 目前暂不支持。 使用 --policy-deny-response=icmp 参数时，请确保您的网络策略允许 ICMP 入站流量。如果 ICMP 流量被入站策略阻止，应用程序将无法收到拒绝通知，并且仍然会遇到连接超时问题
