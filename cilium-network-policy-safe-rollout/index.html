    # Cilium NetworkPolicy 渐进式调试与上线手册

    这份手册是给应用使用方的，不要求你一上来就把策略写对。目标只有一个: 让你先看清真实流量，再一点点把 CiliumNetworkPolicy 收紧到可上线的状态。

    本文档主要基于 Dean Lewis 的 Safe rollout plan 思路，并结合 Cilium 官方文档里对默认拒绝、enableDefaultDeny、Policy Audit Mode 的说明整理而成。不同 Cilium 版本的字段和命令可能略有差异，真正落地前请以你集群当前版本文档为准。

    配套模板见: docs/cilium-network-policy-templates.md

    # 这份手册解决什么问题

    适合下面这类场景:

    • 你知道应用要通信，但还不知道它到底依赖了哪些内部服务、DNS、外部域名或网段。
    • 你怕一上策略就把服务打挂。
    • 你需要给业务方一个可以照着排查的流程，而不是一堆抽象概念。

    不适合下面这类场景:

    • 你想直接全局一键默认拒绝，然后让大家自己救火。
    • 你没有 Hubble，也拿不到 kubectl 查看 CiliumEndpoint 或进入 cilium-agent。

    # 先记住 4 个关键事实

    1. Cilium 默认不是拒绝一切，而是默认放行。只有当某个 Pod 被策略选中，而且该策略里出现了 ingress 或 egress 段时，这个方向才会进入默认拒绝。
    2. 默认拒绝是按方向生效的。只写了 egress，只会收紧出向；只写了 ingress，只会收紧入向。
    3. toFQDNs 依赖 DNS 观测。你没先放行并拦截 DNS，请求域名的规则通常不会按预期工作。
    4. Policy Audit Mode 会把“本来该被拒绝”的流量记录成审计事件，但不真正拦截。它适合摸底，不适合长期生产兜底。

    # 上线总原则

    顺序固定，不建议跳步:

    1. 先观察真实流量。
    2. 先做最保守的基线放行。
    3. 先补平台依赖，再补业务依赖。
    4. 每次只新增一小条策略。
    5. 每加一条都马上验证。
    6. 最后才进入默认拒绝。

    # Step 0: 准备信息

    开始前，先把这几个信息补齐:

    • 目标命名空间: 例如 payments
    • 目标工作负载标签: 例如 app=api
    • 入口调用方: 哪些 Pod 或命名空间会访问它
    • 出口依赖: DNS、kube-apiserver、数据库、消息队列、外部域名、外部 CIDR
    • 业务验证方式: 例如 curl、健康检查、集成测试、页面点击路径

    如果这些都不清楚，就先不要写精细策略，先做观察。

    # Step 1: 用 Hubble 先看真实流量

    先在目标命名空间观察最近流量:

    1

    hubble observe --namespace <namespace> --last 100 -t policy-verdict

    只看被拒绝流量:

    1

    hubble observe --namespace <namespace> --verdict DROPPED

    看最近 5 分钟原始 JSON:

    1

    hubble observe --namespace <namespace> --since 5m -o json

    如果你已经知道某个 Pod 有问题，直接盯它:

    1

    hubble observe --from-pod <namespace>/<pod> -o json

    观察时至少记下这些内容:

    • 谁在访问谁
    • 是 ingress 还是 egress
    • 目标端口是多少
    • 是否有 DNS 查询
    • 是否访问了外部域名或外部 IP
    • Hubble 里显示的是 ALLOWED、DENIED 还是 AUDITED

    判定标准:

    • 你能列出这个应用的入向调用方和出向依赖。
    • 你能区分哪些是业务流量，哪些是平台流量。

    # Step 2: 先建立命名空间内互通的基线

    刚开始做策略时，最容易误伤的是同命名空间里的服务间调用。先用一个“命名空间内互通”策略保住当前行为，再慢慢收紧。

    模板见: docs/cilium-network-policy-templates.md

    你需要的效果是:

    • 同一命名空间里的 Pod 之间还能互相访问。
    • 你开始引入策略后，不会立刻打断应用内部调用。

    判定标准:

    • 同命名空间常见服务调用仍然正常。
    • Hubble 中同命名空间流量仍为 ALLOWED。

    # Step 3: 补齐平台依赖

    业务策略经常不是先死在业务调用，而是先死在平台依赖。最常见的是下面两个:

    • DNS 到 kube-dns 或 coredns
    • 访问 Kubernetes API Server

    # 3.1 先确认你的 DNS Pod 标签

    不同发行版标签不一定完全一致，不要盲抄:

    1

    kubectl -n kube-system get pods --show-labels | grep -E 'dns|coredns'

    多数集群里你会看到类似:

    • k8s-app=kube-dns
    • 或 k8s-app=coredns
    • 在 Cilium 官方示例里，Kubernetes 标签在策略中常表现为带前缀的键，例如 k8s:io.kubernetes.pod.namespace、k8s:k8s-app。如果你照抄旧文章里的无前缀写法不生效，优先以当前版本官方文档和实际 Endpoint 标签为准。

    # 3.2 放行 DNS

    如果你后面要用 toFQDNs，这一步基本是必做项。建议同时允许 UDP/TCP 53。

    # 3.3 放行 kube-apiserver

    不要优先写死 API Server IP，优先使用 toEntities: kube-apiserver。

    判定标准:

    • 应用仍能解析域名。
    • 依赖 leader election、watch、lease、探针的组件没有异常。
    • Hubble 中对 DNS 和 kube-apiserver 的访问是 ALLOWED 或 AUDITED，而不是 DENIED。

    # Step 4: 一次只加一条业务放行

    从这里开始，才进入真正的业务策略收敛阶段。

    推荐顺序:

    1. 先放行最核心的调用链。
    2. 再放行必要的内部服务。
    3. 再放行外部域名或外部网段。
    4. 最后再考虑 L7 限制。

    每次只做一件事，例如:

    • 只允许 web 访问 api:8080
    • 只允许 api 访问 redis:6379
    • 只允许 worker 出向访问 https://api.example.com
    • 只允许 syncer 访问 203.0.113.0/24

    每加一条规则，立刻执行一次业务验证，并同时看 Hubble:

    1

    hubble observe --namespace <namespace> --last 50 -t policy-verdict

    如果验证通过，再继续下一条。不要把五六个依赖一次性全塞进去。

    # Step 5: 必要时用 Audit Mode 摸底

    如果你已经准备开始“默认拒绝”，但又不敢真的拦，可以短时间开启 Policy Audit Mode。

    官方推荐两种方式:

    • 整个 Cilium Daemon 打开审计模式
    • 只对单个 Endpoint 打开审计模式

    生产里更推荐后者，因为影响面小。

    # 5.1 对单个 Pod 开启 Audit Mode

    先拿到 Pod 对应的节点、CiliumEndpoint ID、所在节点的 Cilium Pod:

    1
2
3
4
5

    PODNAME=<pod-name>
NAMESPACE=<namespace>
NODENAME=$(kubectl -n "$NAMESPACE" get pod "$PODNAME" -o jsonpath='{.spec.nodeName}')
ENDPOINT=$(kubectl -n "$NAMESPACE" get cep "$PODNAME" -o jsonpath='{.status.id}')
CILIUM_POD=$(kubectl -n kube-system get pod --field-selector spec.nodeName="$NODENAME" -l k8s-app=cilium -o jsonpath='{.items[0].metadata.name}')

    如果这一步拿不到 CILIUM_POD，先手工检查 Cilium DaemonSet Pod 的真实标签:

    1

    kubectl -n kube-system get pods --show-labels | grep cilium

    开启审计模式:

    1
2

    kubectl -n kube-system exec "$CILIUM_POD" -c cilium-agent -- \
  cilium-dbg endpoint config "$ENDPOINT" PolicyAuditMode=Enabled

    如果镜像里没有 cilium-dbg，请改用 cilium。

    检查是否开启成功:

    1
2

    kubectl -n kube-system exec "$CILIUM_POD" -c cilium-agent -- \
  cilium-dbg endpoint get "$ENDPOINT" -o jsonpath='{[0].spec.options.PolicyAuditMode}'

    你应该看到:

    1

    Enabled

    # 5.2 观察审计结果

    1

    hubble observe --namespace <namespace> -t policy-verdict

    如果看到 AUDITED，说明:

    • 这条流量在真实拦截模式下本来会被拒绝
    • 但因为当前是 Audit Mode，所以被放过去了

    这时就把缺的放行规则补上。

    # 5.3 关闭 Audit Mode

    1
2

    kubectl -n kube-system exec "$CILIUM_POD" -c cilium-agent -- \
  cilium-dbg endpoint config "$ENDPOINT" PolicyAuditMode=Disabled

    注意:

    • Endpoint 级别的 PolicyAuditMode 不是持久化配置，cilium-agent 重启后会恢复成 Daemon 默认状态。
    • 整个 Daemon 的 Audit Mode 会导致该 Daemon 管理的流量不执行网络策略拦截，不建议长期开着。

    # Step 6: 切到真正的默认拒绝

    当你已经满足下面条件时，才能真正进入默认拒绝:

    • 入向调用方已经列全
    • 出向依赖已经列全
    • DNS 已经验证
    • kube-apiserver 已经验证
    • 业务主要路径已经回归通过
    • Hubble 里没有新的 AUDITED 或意外 DENIED

    此时你有两种方式进入默认拒绝:

    • 使用明确的“默认拒绝”策略
    • 让某个工作负载被带 ingress 或 egress 的策略选中，并只保留你显式允许的通信

    如果你要给业务方一个最容易理解的标准动作，建议直接用显式默认拒绝模板，然后一条条补 allow。

    模板见: docs/cilium-network-policy-templates.md

    验证命令:

    1

    hubble observe --namespace <namespace> --verdict DROPPED

    判定标准:

    • 不需要的流量已经开始掉。
    • 业务必需流量仍然 ALLOWED。

    # Step 7: 只在确实需要时再上 L7

    L7 规则适合这些场景:

    • 只允许某个 HTTP 方法
    • 只允许某个 HTTP 路径
    • 需要 DNS 规则支撑 FQDN 访问

    不适合这些场景:

    • 只是想限制来源 Pod 和目标端口
    • 目标端口根本不是 HTTP
    • 你还没把 L3/L4 关系梳理清楚

    几个重点:

    • 大部分流量优先停留在 L3/L4。
    • L7 会引入代理解析，复杂度更高。
    • enableDefaultDeny: false 不能替你规避所有 L7 带来的拦截风险。官方文档明确说明，这个字段不适用于 L7 规则。

    # 常见排查路径

    # 现象 1: FQDN 规则写了，但域名访问还是不通

    先查:

    • 是否已经允许到 DNS
    • 是否是 TCP/UDP 53 都考虑到了
    • DNS Pod 标签是否抄错
    • Hubble 里能不能看到 DNS 查询

    # 现象 2: 明明写了允许策略，还是被拒绝

    先查:

    • endpointSelector 是否选中了正确 Pod
    • 来源标签是不是写错 key 了
    • 跨 namespace 通信时，是否显式匹配了来源 namespace 标签
    • 端口和协议是不是对应得上
    • 是 ingress 被拒绝还是 egress 被拒绝
    • 是否有 egressDeny 或 ingressDeny 优先生效

    # 现象 3: 一上策略健康检查就挂

    先查:

    • 探针是不是访问了未放行端口
    • 应用是否依赖 kube-apiserver
    • 是否有 sidecar、agent、指标采集、日志采集流量没有补齐

    # 现象 4: Hubble 里看到 AUDITED

    说明你当前在审计模式下，真实拦截时这条流量会失败。不要忽略它，要补规则。

    # 现象 5: 不知道 Cilium 现在到底认为这个 Pod 生效了什么策略

    如果有权限，查看 CiliumEndpoint:

    1

    kubectl -n <namespace> get cep <pod-name> -o yaml

    或者进入对应节点的 cilium-agent:

    1

    kubectl -n kube-system exec -it <cilium-pod> -c cilium-agent -- cilium-dbg endpoint get <endpoint-id>

    重点看:

    • Endpoint 身上的真实标签
    • Identity
    • 当前策略是否是 ingress、egress 或 both
    • 被允许或拒绝的是哪些身份、端口和方向

    # 给业务方的最小执行清单

    如果你要把流程进一步简化给一线用户，可以让他们严格按下面执行:

    1. 先观察 10 到 30 分钟真实流量。
    2. 先上命名空间内互通基线。
    3. 先补 DNS。
    4. 再补 kube-apiserver。
    5. 一次只放行一个业务依赖。
    6. 每次改完都做业务验证和 Hubble 验证。
    7. 必要时先开单 Pod Audit Mode。
    8. 没有新的 AUDITED 后，再切默认拒绝。
