# Cilium BGP 在 Leaf-Spine 多 Leaf 场景下的路由设计

# 场景

当前集群节点分布在两个 Leaf 下，地址修正为 10.1.x.x：

  • Leaf1：10.1.5.1
  • 节点：10.1.5.11、10.1.5.12
  • Leaf2：10.1.8.1
  • 节点：10.1.8.13、10.1.8.14

网络拓扑如下：

1
2
3
4
5
6

                         +----------- spine -----------+
                         /                             \
                  10.1.5.1                               10.1.8.1
                   (leaf)                                 (leaf)
                  /      \                               /      \
          10.1.5.11    10.1.5.12                 10.1.8.13    10.1.8.14

假设每个节点各自承载一个 PodCIDR：

  • 10.1.5.11 -> 10.244.11.0/24
  • 10.1.5.12 -> 10.244.12.0/24
  • 10.1.8.13 -> 10.244.13.0/24
  • 10.1.8.14 -> 10.244.14.0/24

本文讨论的是：在使用 Cilium BGP Control Plane 的前提下，跨多个 Leaf 的 Kubernetes 集群应如何设计路由与 BGP 邻居关系。

# 结论摘要

  • 节点与本地 Leaf 建 BGP 邻居更合理，不建议 4 个节点彼此互连建立 BGP。
  • 在 Leaf-Spine 场景下，通常优先推荐 eBGP 的 节点 <-> 本地 Leaf 模式。
  • 如果节点默认路由指向本地业务 Leaf，那么最符合 Cilium 官方边界的方案是“默认路由模型”。
  • 只使用 Cilium 自带的 BGP Control Plane 时，不应假设它会把从对端学到的路由自动导入本机 Linux 路由表。
  • 如果希望每台节点都安装全量 PodCIDR 明细路由，需要额外的 BGP daemon，例如 FRR/BIRD，而不是只依赖 Cilium BGP CP。

# BGP 应该建成 iBGP 还是 eBGP

# 推荐方案

优先考虑：

  • 10.1.5.11 <-> 10.1.5.1
  • 10.1.5.12 <-> 10.1.5.1
  • 10.1.8.13 <-> 10.1.8.1
  • 10.1.8.14 <-> 10.1.8.1

也就是每个节点只和所在机架的本地 Leaf 建立 BGP 邻居。

# 为什么更偏向 eBGP

  • eBGP 更符合常见数据中心 ToR/Leaf 接入模型。
  • 设计上更清晰，节点和 Leaf 通过不同 ASN 建邻，Leaf/Spine 再按网络团队既有规范继续传播路由。
  • iBGP 会带来经典限制：从一个 iBGP 邻居学到的路由默认不会再转发给另一个 iBGP 邻居，因此通常需要 full-mesh 或 route reflector。
  • 对于 Kubernetes Node 接入 ToR 这种场景，eBGP 往往更直接，维护成本也更低。

# 什么时候 iBGP 也可以

如果网络侧明确要求节点与 Leaf 处于同一 AS，iBGP 当然也能工作，但前提是：

  • Leaf/Spine 内部已经有完整的 iBGP 传播设计。
  • 已经规划好 route reflector 或 full-mesh。
  • 你清楚 next-hop、路由反射、策略控制这些网络侧前提。

如果没有这些前提，优先用 eBGP 更稳妥。

# 模型 1：默认路由指向本地业务 Leaf

这是更推荐的模式，也是最符合 Cilium 官方文档边界的方式。

这个模型的核心是：

  • 节点不需要知道所有远端 PodCIDR 的明细去向。
  • 节点只需要把未知远端 Pod 流量交给本地业务 Leaf。
  • 由 Leaf/Spine 负责全网收敛。

# 模型 2：每台节点都安装全量 PodCIDR 路由

这个模型也是可行的，但不能只靠 Cilium BGP Control Plane。

例如在 10.1.5.11 节点上，理论上你可能希望看到如下本机路由：

1
2
3

- 10.244.12.0/24 via 10.1.5.12
- 10.244.13.0/24 via 10.1.5.1
- 10.244.14.0/24 via 10.1.5.1

这样一来，节点本机就“知道每段 PodCIDR 应该交给谁”。

但这里要注意：Cilium 官方文档里提到的“每个节点都知道所有 PodCIDR，并把路由写入 Linux 内核路由表”，需要额外的路由组件来完成，典型是：

  • FRR
  • BIRD
  • 其他能够把 BGP 学到的路由写入本机 FIB 的 daemon

# Cilium 会不会把对端发来的路由导入到本机

短结论：默认不要这样理解。

更准确地说：

  • Cilium BGP Control Plane 负责 BGP 控制面能力，例如建立邻居、通告 PodCIDR、通告 Service VIP。
  • 但 Cilium 官方明确说明，BGP Control Plane 不负责直接编程数据平面，不应用它来建立集群内部节点可达性。
  • 这意味着它不应被理解为一个“把收到的 BGP 路由自动写进节点内核路由表”的通用路由守护进程。

因此，在只使用 Cilium BGP CP 的情况下：

  • 节点通常不会自动获得远端所有 PodCIDR 的明细 ip route
  • 更合理的预期是：节点向外宣告自己的前缀，由网络设备承担全网传播和转发职责

# PodCIDR 广告的边界

# Cilium 只会通告本节点持有的 PodCIDR

在 Cilium 原生 BGP 模式下，PodCIDR 通告是按节点本地前缀进行的。

结合本文示例：

  • 10.1.5.11 只会通告 10.244.11.0/24
  • 10.1.5.12 只会通告 10.244.12.0/24
  • 10.1.8.13 只会通告 10.244.13.0/24
  • 10.1.8.14 只会通告 10.244.14.0/24

一个节点不会默认替别的节点代为起源和通告 PodCIDR。

# 如果某个节点没有建立 BGP 邻居，会发生什么

如果某个节点没有对外建立 BGP 会话，或者没有被对应的 CiliumBGPClusterConfig 选中，那么：

  • 该节点上的 PodCIDR 不会被 Cilium 对外通告
  • 外部网络也就学不到这段 PodCIDR 的路由
  • 结果是该节点上的 Pod 可能无法被外部网络直接访问

这也是为什么在 native routing + PodCIDR advertisement 模式下，通常要求每个承载 Pod 的工作节点都能与本地 Leaf 正常建邻。

# 默认路由如果指向本地业务 Leaf，会怎样

这是推荐情况。

只要满足下面两个前提，整体就能正常工作：

  • 节点默认路由指向本地业务 Leaf
  • Leaf/Spine 已经正确学习并传播所有 PodCIDR 和需要通告的 Service 路由

此时：

  • 节点侧仅需保留本地 PodCIDR 路由和默认路由
  • 远端 Pod 访问依赖 Leaf/Spine 的三层转发
  • 这是最符合 Cilium 在 native routing 模式下的推荐边界的做法

# 默认路由如果不在业务 Leaf，而在管理 Leaf，会怎样

这是需要特别警惕的情况。

如果默认路由不指向业务 Leaf，而指向另外一个管理 Leaf：

  • 如果启用了 DefaultGateway auto-discovery，Cilium 会倾向于把默认网关识别为 BGP peer
  • 那么建立 BGP 邻居的对象可能会变成管理 Leaf，而不是业务 Leaf
  • 节点的远端 Pod 流量也会优先按 Linux 默认路由走向管理 Leaf

如果管理 Leaf 没有承载这些 PodCIDR，或者没有参与业务网络转发，就会出现：

  • 路由绕行
  • 不必要的跨域转发
  • 甚至黑洞

因此有两条实践建议：

  • 最好让节点默认路由直接指向业务 Leaf
  • 如果默认路由无法改动，就不要依赖 DefaultGateway auto-discovery，而应手工指定 peerAddress 指向业务 Leaf

如果对端不是直连 eBGP 邻居，还需要结合实际路径评估是否开启 ebgpMultihop。

# Cilium 原生 BGP 和 FRR+BGP 怎么选

# Cilium 原生 BGP 更适合什么

更适合下面这些目标：

  • 希望通过 Cilium CRD 统一配置 BGP
  • 主要需求是对外通告 PodCIDR 或 Service VIP
  • 节点默认路由已经指向本地业务 Leaf
  • 不要求节点把学到的远端路由装进本机 Linux FIB
  • 希望运维面尽量简单

这种模式下，最典型的路径是：

  • 每个节点对本地 Leaf 建 eBGP
  • 节点通告本机 PodCIDR
  • Leaf/Spine 负责全网传播这些前缀

# FRR+BGP 更适合什么

更适合下面这些目标：

  • 希望节点本机安装全量 PodCIDR 路由
  • 需要把学到的 BGP 路由写入 Linux 内核路由表
  • 需要更完整的 BGP 策略控制
  • 希望节点本身更接近一个完整路由器角色

FRR 的价值在于：

  • bgpd 负责 BGP 邻居和路由学习
  • zebra 负责把 FIB 写入 Linux 内核路由表

# 外置 BGP daemon 现在是不是已经废弃

这里要精确区分两件事：

  • Cilium 官方旧文档里的 Using BIRD to run BGP 和 Using Kube-Router to Run BGP 已经是 deprecated
  • 但“节点上同时运行其他 BGP daemon，例如 FRR/BIRD”这种架构思路，本身并没有被禁止

换句话说：

  • 官方主推路线已经是 Cilium BGP Control Plane
  • 如果你需要更完整的主机路由器能力，仍然可以引入 FRR，只是这不再是 Cilium 官方最推荐的主路径

# 跨 Leaf 集群流量互通时怎么选

如果目标是“同一个 Kubernetes 集群跨多个 Leaf 正常互通”，建议按需求分成两类：

  • 如果优先追求简单和稳妥，建议使用 Overlay 承担集群内部 east-west 流量，再用 BGP 对外通告 Service VIP
  • 如果明确要求 PodCIDR 在 underlay 中直达，建议使用 native routing + Cilium 原生 BGP，并确保每个工作节点都与本地 Leaf 建立 BGP

如果你的诉求是：

  • 让每个节点自己学到所有远端 PodCIDR
  • 节点本机直接拥有明细路由
  • 节点具备更完整的 BGP/FIB 行为

那么应转向 FRR+BGP 这类方案，而不是只依赖 Cilium 原生 BGP。

# Overlay 模式下还能不能宣告 PodCIDR

可以，但要分清楚“能不能”和“有没有必要”。

# 能不能

Overlay 模式下，Cilium 依然可以通过 BGP 对外通告 PodCIDR。BGP 通告的边界仍然没变：

  • 只会通告本节点持有的 PodCIDR
  • 不会自动代替其他节点通告前缀

# 有没有必要

Overlay 模式本身已经解决了集群内部跨 Leaf 的 east-west 流量，因为内部 Pod-to-Pod 转发走的是封装隧道。

因此在 Overlay 模式下继续通告 PodCIDR，通常不是为了解决集群内部互通，而是为了：

  • 让机房现网可以直接访问 PodIP
  • 让外部网络知道去某个 PodCIDR 应该走哪个节点

所以 Overlay + BGP PodCIDR Advertisement 更像是：

  • Overlay 解决内部互通
  • BGP 解决外部网络到 PodIP 的路由可达性

# 回程路由和 masquerade 为什么会出问题

这是 Overlay + PodCIDR advertisement 场景下最容易忽略的点。

# 问题 1：回程路由是否真的存在

假设：

  • 外部客户端：172.16.10.20
  • Pod：10.244.11.23
  • Pod 所在节点：10.1.5.11

如果 BGP 已经把 10.244.11.0/24 广告给现网，那么外部访问 Pod 的去程可能是通的：

1

172.16.10.20 -> 10.244.11.23

但去程通不代表回程一定通。

Pod 回包时如果保留源地址：

1

10.244.11.23 -> 172.16.10.20

那么回程链路上必须满足：

  • 机房网络完整接受并传播 PodCIDR
  • 回程路径上的设备认识 10.244.11.0/24
  • 中间防火墙/ACL 允许 Pod 网段往返

只要其中任何一段不认识 PodCIDR，就可能出现：

  • 外部请求能到 Pod
  • Pod 回包却在中途黑洞

# 问题 2：masquerade 会不会把 PodIP 改掉

Cilium 默认会对离开集群的 Pod 流量做 masquerade。也就是说，Pod 发往外部网络的回包可能会被 SNAT 成节点 IP。

例如：

1

172.16.10.20 -> 10.244.11.23

Pod 回包时可能变成：

1

10.1.5.11 -> 172.16.10.20

而不是：

1

10.244.11.23 -> 172.16.10.20

对于“外部直接访问 PodIP”这个目标来说，这通常不是你想要的结果，因为客户端访问的是 PodIP，回来的却是节点 IP。

# 正确理解 no-SNAT 的边界

这里不能简单粗暴地把 masquerade 全关掉。

因为如果全部关闭：

  • Pod 访问那些并不知道 PodCIDR 的外部网络时
  • 对端可能无法把回包送回 PodIP

更合理的方式通常是：

  • 对确实能够回 PodCIDR 的机房内部网段，配置不做 masquerade
  • 对互联网或其他不认识 PodCIDR 的网络，继续保留 masquerade

因此在 Overlay + BGP PodCIDR 场景里，更稳妥的做法通常是：

  • Overlay 承担集群内部 east-west
  • BGP 把 PodCIDR 广告给机房内部真正需要访问 PodIP 的网络
  • 对这些内部网段配置 selective no-SNAT
  • 对其他外部网络继续维持默认的 masquerade 行为

# 结合本场景的最终建议

# 推荐拓扑

1

10.1.5.11/12 default via 10.1.5.1
10.1.8.13/14 default via 10.1.8.1

Node <-> Local Leaf 建立 eBGP
Leaf/Spine 负责传播 PodCIDR / Service VIP

# 不建议的误区

  • 不要默认认为 Cilium 会把收到的 BGP 路由自动导入本机 ip route
  • 不要在默认路由指向管理 Leaf 的情况下直接使用 DefaultGateway auto-discovery
  • 不要把“Cilium 能通告路由”误解为“Cilium 自动承担了完整节点路由器的职责”

# 额外说明

如果集群运行的是 Overlay 模式，例如 VXLAN 或 Geneve，那么 BGP 的主要作用更偏向于对外宣告 PodCIDR 或 Service VIP，而不是替代集群内部隧道转发逻辑。本文更适用于 native routing 语义下的讨论。
