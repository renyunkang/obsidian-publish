    # 调试使用：

    # 进入调试模式的方法：

    • dlv attach pid：类似与gdb attach pid，可以对正在运行的进程直接进行调试(pid为进程号)。
    • dlv run|debug：run命令已被debug命令取代，运行dlv debug test.go会先编译go源文件，同时执行attach命令进入调试模式，该命令会在当前目录下生成一个名为debug的可执行二进制文件，退出调试模式会自动被删除。
    • dlv exec executable_file：直接从二进制文件启动调试模式。
    • dlv core executable_file core_file：以core文件启动调试，通常进行dlv的目的就是为了找出可执行文件core的原因，通过core文件可直接找出具体进程异常的信息。

    # dlv传递参数

    添加**–**来告诉dlv所有后续参数都应逐字传递到二进制文件，而无需尝试解析它们: dlv exec /mypath/binary -- --config=config.yaml

    # dlv调试常用命令

    args打印函数的参数
    locals查看函数的局部变量
    continuec继续执行到设置的断点
    nextn单步执行
    break main.go:90b在main.go文件的第10行设置断点
    breakpointsbp打印出所有的断点信息
    clear清除单个断点
    clearall清除所有断点
    on当运行到某断点时执行相应命令
    condition有条件的中断断点 condition 1 number==20
    stacks打印调用堆栈信息
    goroutinegr查看当前Goroutine相关的信息，加上数字切换指定ID的routtinue
    goroutinesgrs查看当前Goroutine相关的信息
    backtracebt打印当前goroutine堆栈信息
    frame + IDf切换指定IDframe
    list + 行数l查看指定行数上下10行源代码【默认当前所运行处】
    printp打印变量值
    trace类似于打断点，但程序运行到该处时不会中断而是继续执行，同时会输出一行提示信息。
    call修改变量的值 call number = 12
    help打印帮助信息
